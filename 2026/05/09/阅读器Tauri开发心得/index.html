阅读器 Tauri 桌面版开发心得

发表于 2026-05-09 分类于 技术

上次写完阅读器的设计思路之后，我第二天就开始动手做桌面版了。

为什么一定要做桌面版？原因特别简单——我想让同学双击 exe 就能用。

我原来的 web 版是这样的：后端 Express + Axios + Cheerio，前端 React，整套跑在浏览器里，我还顺手配了 Docker 部署。自己用完全没问题，但要是发给不写代码的同学，我得先教他们装 Node.js，或者教他们装 Docker，然后还得打开终端敲命令。

这不现实。我一个舍友连压缩包都不会解。

所以必须打包成 exe。双击运行，别的什么都不用管。

Electron 和 Tauri 之间怎么选

能做桌面端 exe 的框架，我第一个想到的肯定是 Electron。但 Electron 有个让我没法接受的问题——体积太大。一个什么都没写的 hello world 打包出来就上百 MB。

为什么？因为 Electron 给每个应用都塞了一个完整的 Chromium 浏览器进去。

Tauri v2 不一样。它不塞浏览器，而是直接调用 Windows 系统自带的 WebView2。前端还是用 React 写，但后端从 Node.js 换成了 Rust。最终打包出来的 exe 只有 14MB，不到 Electron 的十分之一。

14MB 什么概念？QQ 安装包快 200MB 了。一个带三个书源的桌面阅读器，体积只有 QQ 的百分之七。

但 Tauri 有个门槛：后端要用 Rust 重写。

从零开始写 Rust

我之前完全没碰过 Rust。只说听说过，知道它是”系统级语言”、”安全”、”难学”。

书源的逻辑全在后端——搜索、抓取目录、解析章节内容。原来在 Node.js 里用 Axios 发 HTTP 请求，用 Cheerio 解析 HTML，用 iconv-lite 处理 GBK 编码。

到了 Rust 这边，每一个东西都得找对应的库：Axios 换成 reqwest，Cheerio 换成 scraper，iconv-lite 换成 encoding_rs。连 URL 编码都得单独装一个叫 urlencoding 的包。

Rust 的语法倒不是最难的。所有权、借用、生命周期确实需要适应，但编译器报错极其友好，不仅告诉你怎么错了，还会建议怎么改。很多时候顺着编译器的提示就能把代码改对。

真正折磨人的是 Windows 工具链。

Win11 预览版上的地狱体验

我先装了 Rust 的 GNU 工具链。代码编译通过，生成了 exe。双击运行——

没反应。

不是报错，是双击之后什么都没发生。用命令行看了一下，提示 os error 193，意思是”不是有效的 Win32 应用程序”。文件格式明明是对的，但 Windows 就是不认。

只能换成 MSVC 工具链。然后问题又来了：Git 自带的 link.exe 和 MSVC 的链接器重名了，系统优先找到了 Git 的那个。link.exe 是 Unix 下创建符号链接的工具，根本不是用来链接程序的。链接的时候报错”extra operand”，我盯着看了五分钟才反应过来——它用的不是微软的链接器。

把 Visual Studio 的 C++ 构建工具装上之后，终于能正常编译了。

这些跟写代码一点关系都没有。但就是这些不相干的东西，占了我差不多一半的开发时间。

三个书源的 Rust 实现

书源本身的设计和 web 版完全一样，每个书源实现三个方法：search_books、get_chapters、get_chapter_content。由一个路由器（mod.rs）根据用户选择的书源来分发调用。

光遇 API 最省心。它是 JSON 接口，数据格式规整，reqwest 拿到响应直接反序列化就完了。但它有个毛病：API 服务器不稳定，经常某个域名挂了。我设了七个域名做 fallback，一个超时自动切下一个。用了一个宏来减少重复代码，但宏在 Rust 里跟函数不一样，变量的生命周期问题调了好久。

笔趣阁最折腾。它用的是 GBK 编码的 HTML，搜索还是 POST 表单。Node.js 里一套 fetch + iconv-lite 就能搞定的事，Rust 这边要先把中文关键词编码成 GBK 字节，拼到表单里发 POST，收到响应再把 GBK 字节解码回来。而且正则的处理也比 Node.js 严格——Rust 的 regex 不支持反向引用，之前用来匹配成对 HTML 标签的正则 <(script).*?</\1> 直接报错，得改成显式的标签名。

七星阁和笔趣阁整体逻辑差不多，但它是 UTF-8 编码，省了编解码这一步。不过章节内容里的脏数据很多，什么”本章未完”、”第(1/3)页”、分类标签，全得用正则一层一层清。正则写多了之后发现，清理数据比抓取数据写的代码还多。

前后端通信的变化

web 版的前后端通信就是 fetch(‘/api/…’)，标准 HTTP 请求。

Tauri 版变成了 invoke(‘search_books’, { keyword, source })，走的是 Tauri 的 IPC。前端用 TypeScript 封装了 invoke 调用，接口签名基本没变，但底层从 HTTP 变成了进程间通信。

这个改动对前端来说几乎是透明的。除了把 fetch 换成 invoke，其他逻辑一行没改。这也是 Tauri 设计得好的地方——它让前端开发者几乎感知不到后端的切换。

最大的感受：最难的不是写代码

整个项目做完，我最大的感受不是”Rust 真难”也不是”Tauri 真香”——而是”把软件交到别人手里，比写软件本身难多了”。

功能我在 web 版就已经写好了。搜索、目录、阅读器、日夜间模式，web 版全都有。

但为了让别人能双击运行，我做的事情包括：把后端从 Node.js 迁移到 Rust、折腾 Windows 工具链、处理三个不同书源的各种编码和格式、配置 GitHub Actions 自动构建、写面向小白用户的图文 README、在 GitHub Releases 上发布 exe。

这些全都不是”功能开发”，但每一项都不可或缺。没有它们，软件就只是我电脑上的一个 localhost:3000，别人根本看不见。

我以前觉得软件工程最难的是写代码。现在发现，代码只是冰山在水面上的那一角。水面之下是分发、部署、文档、兼容性——这些才是决定一个软件能不能真正”存在”的东西。

现在的状态和后面要做的事

目前 v0.1.0 已经发在 GitHub Releases 上了，14MB 的单 exe，下载双击就能用。三个书源都能正常搜索和阅读。

但说实话，离”好用”还有距离。接下来几天我会集中做几件事：

  1. 书源扩展。现有的三个书源覆盖率太低了，搜一些冷门书基本搜不到。得再加几个源，至少要覆盖到常见的网文。

  2. 书架和历史记录。现在每次打开软件都要重新搜索，没有任何持久化。下一步会用 SQLite 把用户的阅读进度、书架收藏存到本地。

  3. 阅读体验优化。现在的字号只有三档，翻页也没有动画。至少要加上自定义字号、行距、翻页动画。

等这些做完，v1.0 才算真的能用。
